## 🎯 Mission

Ship a free-forever, **fully client-side**, browser-only floating sticky-notes app that lives in a real OS-level floating window via Document Picture-in-Picture API. **No install. No signup. No cloud.** Notes stored in OPFS — they never leave your device.

Win by being the **only browser-native floating sticky-notes** in existence (validated: zero direct competitors, only OS-native installs like Stickies.app + Windows Sticky Notes + Notezilla).

## 🏗️ Stack (MIRRORS FTN EXACTLY)

| Layer | Choice | Why |
|---|---|---|
| Framework | **Astro 6** MPA | SEO-first static landing + light interactivity |
| Styling | **Tailwind v4** + `@theme` in `global.css` | NO `tailwind.config.js` |
| Language | **TypeScript strict** | No `any` in shipped code |
| Editor | **Tiptap v2** + extensions | Notion-grade UX, off-the-shelf |
| Storage | **OPFS** (Origin Private File System) | Local-first, ~1GB capacity |
| Sync (cross-tab) | **BroadcastChannel API** | Same-origin, no server |
| Floating window | **Document Picture-in-Picture API** | Chrome/Edge 116+ |
| Deploy | **Cloudflare Pages** auto from `main` push | Zero infra cost |
| Domain | **Cloudflare Registrar** | At-cost forever, one-click custom domain |
| Brand assets | Generate via `scripts/gen-icons.mjs` + `gen-og-image.mjs` | Sharp + inline SVG |
| Icons | **Inline Lucide SVG paths** in component data | Zero icon-lib dep |

**NO backend.** Pure static site + browser APIs. Zero Cloudflare Worker needed for v0.1.

### ❓ DECISION 1 — Brand color: yellow vs other

Sticky notes scream YELLOW universally. But options:

| Option | Accent | Vibe | Differentiation |
|---|---|---|---|
| ⭐ **A: Warm yellow** (`#facc15` / `#fde047`) | Yellow | Classic sticky-note feel | Distinct from LiveCaptionIt blue |
| B: LiveCaptionIt blue match (`#0070f3` / `#3291ff`) | Blue | Portfolio consistency | Sibling product visual link |
| C: Custom (specify) | Your call | — | — |

**My pick:** A. Yellow IS sticky notes. Brand recall is everything. The "Document PiP Family" portfolio link can be communicated through nav cross-linking + footer, not color.

---

### ❓ DECISION 2 — Editor library

| Option | Bundle | Capability | Risk |
|---|---|---|---|
| ⭐ **A: Tiptap v2 + 4 extensions** | ~80KB gzip | Notion-grade: rich text, checkboxes, colors, links, Markdown shortcuts | Low — battle-tested, big community |
| B: ProseMirror (raw) | ~150KB | Maximum control | High — verbose API, longer ship time |
| C: Hand-rolled contentEditable | ~5KB | Lightweight | Very high — selection bugs, cross-browser pain (TipTap exists for a reason) |
| D: Lexical (Meta) | ~40KB | Modern, React-friendly | Medium — less Astro-friendly, smaller community |

**My pick:** A. Per your firm preference: "off-the-shelf libs > hand-rolled." Tiptap is the "Notion-grade quality" bar you want for editor primitives.

---

### ❓ DECISION 3 — OPFS storage schema

| Option | Layout | Pros | Cons |
|---|---|---|---|
| ⭐ **A: One JSON file per note + index** | `notes/{noteId}.json` + `notes/_index.json` | Easy CRUD, atomic single-note writes, fast list rendering from index | Index can drift from files (mitigate with recovery sweep on app load) |
| B: Single mega-JSON file | `notes/all.json` | Simplest serialization | Whole-file rewrite on every save = slow at scale; lock contention |
| C: SQLite via wa-sqlite | `notes.db` (with WAL) | Best query performance, full-text search built in | ~600KB WASM payload, more complex |

**My pick:** A. Mirrors how Apple Notes / Bear store notes. Index for list speed, per-note files for atomic writes. SQLite is overkill for v0.1; we can migrate later if needed.

---

### ❓ DECISION 4 — Storage fallback when OPFS unavailable

OPFS isn't supported in: Firefox <111, Safari <17, private/incognito mode (denied), older browsers.

| Option | Fallback | Pros | Cons |
|---|---|---|---|
| ⭐ **A: IndexedDB fallback transparently** | Same API, different backend | Works on all modern browsers | Slightly slower writes, can't be "file-like" |
| B: Show "Browser not supported" message | Hard block | No partial-functionality bugs | Loses ~5% of audience pre-launch |
| C: Fall back to localStorage | 5MB cap | Universal | Notes lost when user exceeds 5MB |

**My pick:** A. The user-experience cost of option B is too high (silent users who just bounce). IndexedDB is the right safety net.

---

### ❓ DECISION 5 — Default theme on first visit

| Option | Behavior | Pros | Cons |
|---|---|---|---|
| ⭐ **A: Match system (`prefers-color-scheme`)** | Dark if user has system dark, else light | Respects OS preference | First-paint flash mitigated by theme-bootstrap script in `<head>` |
| B: Always dark | Forces dark | Distinctive brand | Mismatch with sticky-note metaphor (sticky notes are yellow-on-white IRL) |
| C: Always light | Forces light | Matches IRL sticky notes | Bad for night use (the use case for floating notes during late-night work) |

**My pick:** A. Same as FTN. The theme-bootstrap pattern prevents flash. User can override via toggle (stored in localStorage).

---

### ❓ DECISION 6 — Notes app URL strategy

| Option | URL | Pros | Cons |
|---|---|---|---|
| ⭐ **A: `/app`** | clean | Clear separation: marketing at `/`, tool at `/app` | One extra click |
| B: `/` is the app, `/about` is marketing | inverted | Power-user-friendly | Hurts SEO — marketing copy = ranking content |
| C: Embed app on `/` below the fold | hybrid | One-page experience | Slower initial load (Tiptap bundle eager-loaded) |

**My pick:** A. Same pattern as FTN (`/transfer` is the tool, `/` is marketing). SEO needs marketing copy above the fold. The "Open Notes →" CTA is one click.

---

### ❓ DECISION 7 — Default PiP window size

| Option | Size | Pros | Cons |
|---|---|---|---|
| ⭐ **A: 400×500 px** | Sticky-note proportions | Looks like a giant Post-it | Smaller than typical browser window |
| B: 300×400 px | More compact | Stays out of the way during meetings | Cramped for >100 chars of text |
| C: 500×600 px | Roomy | Comfortable typing | Eats more screen real estate |

**My pick:** A. Matches the sticky-note metaphor visually + per-note size is saved in OPFS, so users can resize and it sticks. A reasonable default + user override = best UX.

---

### ❓ DECISION 8 — License + privacy posture

| Option | License | Privacy claim | Pros | Cons |
|---|---|---|---|---|
| ⭐ **A: MIT, publicly auditable repo on GitHub** | MIT | "Open source, your notes never leave your device" | Maximum trust + viral potential (people share repos) | Anyone can fork |
| B: Source-available (visible, not redistributable) | Custom | Same privacy claim, but harder to clone | Some moat | Not really "open source" — community pushback |
| C: Closed source, private repo | proprietary | "Trust us, your notes are local" | Maximum moat | Lower trust signal, can't link "view source" |

**My pick:** A. Same as FTN + LiveCaptionIt + heicpix. The privacy claim is *only* credible if the code is auditable. Plus indie SaaS doesn't actually have moats from code (clones get out-built by brand + distribution + speed). MIT.
